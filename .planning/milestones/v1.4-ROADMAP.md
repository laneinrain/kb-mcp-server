# Milestone v1.4: Qwen Rerank Search

**Status:** 🚧 PLANNED  
**Phases:** 13–15  
**Total Plans:** 7 (estimated)

## Overview

Improve search precision with two-stage retrieval: Chroma vector recall (`RERANK_CANDIDATES` default 30) followed by CherryIn Qwen3 rerank (`qwen/qwen3-reranker-0.6b`). Applies to REST search and MCP `search_knowledge`. Graceful fallback to vector-only when disabled or API unavailable.

## Phases

### Phase 13: Rerank Client

**Goal:** `@kb/core` exposes `RerankClient` calling CherryIn `/v1/rerank` with retry and typed response mapping.

**Requirements:** RETR-02, RETR-03, RETR-04, RETR-05

**Plans:** 2/2 planned

- [ ] 13-01: `RERANK_MODEL` constant + `RerankClient` (fetch to `{CHERRYIN_BASE_URL}/rerank`)
- [ ] 13-02: Unit tests + optional live test (`skipIf` without `CHERRYIN_API_KEY`)

**Cross-cutting constraints:**
- Same OpenAI-compatible auth header as `EmbeddingClient`
- Cohere/Jina response shape: `{ results: [{ index, relevance_score }] }`
- No new secrets beyond existing CherryIn config

---

### Phase 14: Search Pipeline

**Goal:** `SearchService` implements recall → ACL filter → rerank → topK with fallback.

**Requirements:** RETR-06, RETR-07, RETR-08, RETR-09, RETR-10

**Plans:** 3/3 planned

- [ ] 14-01: Inject `RerankClient` into `SearchService`; recall `RERANK_CANDIDATES` when enabled
- [ ] 14-02: ACL filter before rerank; map rerank scores to `SearchResult.score`
- [ ] 14-03: Fallback on disable/failure; wire through `apps/backend` + `apps/mcp-server` service factories

**Cross-cutting constraints:**
- `topK` max 10 unchanged for final response
- Snippet truncation (500 chars) remains display-only, after rerank
- MCP and REST share one `SearchService` instance

---

**Requirements:** RETR-11, RETR-12, RETR-13

**Plans:** 2/2 planned

## Milestone Summary

**Resolves from prior milestones:**
- RETR-02 cross-encoder reranking (deferred since v1.0)
- Retrieval precision gap on large documents / noisy top-k

**Explicitly deferred:**
- RETR-01 hybrid BM25

**Risk notes:**
- CherryIn `/v1/rerank` endpoint must support `qwen/qwen3-reranker-0.6b` — validate in Phase 13 live test
- Added latency: one rerank API call per search query
- ACL + small candidate pool after filter may reduce rerank effectiveness — document behavior
